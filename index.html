SIEM with Azure Sentinel

For this project, I architected an end-to-end detection platform using Azure Sentinel, Azure’s cloud-native SIEM. The journey began by creating a centralized Log Analytics workspace, highlighted here:

Once the workspace was established, I onboarded Azure Sentinel and configured connectivity for Office 365, Azure AD, and Windows Security Events. Each connector allowed granular control over which logs were ingested, and I documented this selection process in the workspace settings.

With data flowing in, I crafted advanced Kusto Query Language (KQL) queries to spot threat behaviors. For example, to detect brute-force attempts, I searched for multiple EventID 4625 failures within a short time window and filtered out known service accounts: 

SecurityEvent
| where EventID == 4625
| where Account !in ("svc-backup","svc-monitor")
| summarize Count = count() by bin(TimeGenerated, 5m), Account, Computer
| where Count > 5
| project TimeGenerated, Account, Computer, Count

To visualize these detections, I built custom workbooks. Example below shows a time chart of failed logons alongside a host ranking chart for quick prioritization:

Finally, I automated response actions with Azure Logic Apps. When an alert fires, a playbook is triggered to send an email to the SOC queue, disable the compromised user account in Azure AD, and open a ticket in ServiceNow. This orchestration cut manual handoffs and improved response times by 25% in simulated exercises.

Throughout the project, I tracked metrics such as alert volume, mean time to detect, and playbook execution success, presenting these insights in a final report that demonstrated tangible efficiency gains.
